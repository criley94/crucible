# Design: Wave 3 -- Claude Code Integration

**Author:** Dante (TL)
**Status:** Complete -- all decisions implemented
**Date:** 2026-03-16

---

## Context

Wave 3 was proposed as a single integration wave covering agent bootstrap, experience protocol, credential management, and team provisioning. During implementation, the work was split into two Wave 3.5 OpenSpec changes (experience-migration and team-provisioning) plus direct operational work on the CLAUDE.md and agent definition files. This design document consolidates the architectural decisions that were made across all of that work.

---

## Architectural Decisions

### AD1: Credential Storage -- JSON File at ~/.config/teamforge/

**Decision:** Store all credentials in a single JSON file at `~/.config/teamforge/credentials.json`.

**Resolved from:** Proposal B1 (how does Claude Code access secrets?)

The file contains four fields: `api_url`, `api_key`, `org_slug`, `team_slug`. This resolves proposal option (b) -- a path-referenced secrets file. The file is outside the repo, never committed to git, and set to chmod 600.

The JSON format was chosen over environment variables because Claude Code agents can reliably read files via `cat` in bash, and a single file holds all four values without needing multiple env vars. The agent definition instructs agents to suppress API key display in output.

**What was NOT included:** The spec proposed a `default_project_slug` field. This was dropped -- agents do not need a default project slug at bootstrap. The org and team slugs are sufficient for identity loading.

### AD2: Agent Definition Files -- API-Bootstrapped Thin Launchers

**Decision:** Agent definition files are thin launchers that instruct Claude Code to bootstrap identity from the TeamForge API. They contain YAML frontmatter (Claude Code metadata), a bootstrap protocol (credential reading + API calls), an experience protocol, and an operating model section.

**Resolved from:** Proposal B2 (generated vs. hand-maintained?)

The actual implementation chose option (a) -- generated by a Python provisioning script (`scripts/provision_team.py`). This was the opposite of the spec's recommendation (option b, hand-maintained), but proved correct because:
- 10 agents needed files (9 roster + codebase-analyst), not 8
- The bootstrap protocol template is substantial (~80 lines)
- Hand-maintaining 10 identical files is error-prone
- A provisioning script ensures consistency and enables re-provisioning

### AD3: GCP Identity Token -- Dual Authentication

**Decision:** All API calls require both an API key (X-API-Key header) AND a GCP identity token (Authorization: Bearer header).

This was NOT in the original Wave 3 spec. It was discovered during implementation because Cloud Run requires IAM authentication. The GCP org policy blocks `allUsers` and `allAuthenticatedUsers` access at the infrastructure layer, meaning a valid GCP identity token is needed even though `cloudbuild.yaml` uses `--allow-unauthenticated`.

**How it works:** The `--allow-unauthenticated` flag in cloudbuild.yaml tells Cloud Run not to enforce IAM at the service level. However, the GCP org policy overrides this, requiring IAM authentication anyway. The net result is dual-auth: the GCP token satisfies IAM, the API key satisfies application-level auth.

Every agent bootstrap template includes `gcloud auth print-identity-token` as Step 2. This was added to CLAUDE.md, seed.md, and all generated agent definition files.

### AD4: CLAUDE.md as TL Direct-Load Pattern

**Decision:** The project-level `CLAUDE.md` instructs Claude Code to operate AS the Tech Lead directly, not to dispatch the TL as a sub-agent.

The CLAUDE.md file opens with: "You ARE [TL Name], the Tech Lead for Team [Name]. You are NOT an outer agent that dispatches [tl-slug]." This prevents a common failure mode where the main session agent becomes a pass-through relay instead of operating with TL identity.

The TL slug, team slug, and org slug are embedded in the CLAUDE.md template. All other agents are dispatched as sub-agents through the TL using Claude Code's Agent tool.

### AD5: Seed Agent Architecture

**Decision:** A dedicated seed agent (`seed.md`) handles first-time team provisioning. It is a single-purpose agent that provisions and exits.

The seed agent is installed at `~/.config/teamforge/seed.md` via `scripts/install_seed.sh`. It reads the team roster from the API, generates CLAUDE.md and agent files, and reports completion. It does not do development work.

**Alternative invocation:** The same provisioning logic is available as a standalone Python script (`scripts/provision_team.py`) that can be run without launching a Claude Code session. A bash wrapper (`scripts/provision_team.sh`) provides convenience shell invocation.

### AD6: Experience Protocol -- Strengthened Query Triggers

**Decision:** The experience protocol in CLAUDE.md and agent files includes explicit trigger conditions for when to query experience, with the heuristic: "When in doubt, QUERY."

The original spec defined experience retrieval as "on-demand when you need it." Implementation added specific triggers:
- Sponsor asks about preferences, history, or past decisions
- Sponsor asks a question that could have been discussed before
- Making a technical or process decision the team may have encountered before
- Conversation touches team dynamics, relationships, or working style
- Encountering a familiar pattern or needing cross-project context

This was strengthened after observing that agents were under-querying experience in early testing.

### AD7: Experience Capture -- Proactive, Not Deferred

**Decision:** Agents write experience proactively during sessions, not only at session close.

The bootstrap template explicitly states: "Write proactively -- do not wait for session close. If the session ends abruptly, unwritten observations are lost." This resolves proposal NB3 in favor of immediate writes.

### AD8: Agent File Inventory -- Full Roster + Codebase Analyst

**Decision:** Generate an agent definition file for every roster member, including James (PO).

The original spec excluded James because "there is no dedicated PO on this team." The provisioning script generates files for ALL roster members because:
- The roster comes from the database, not from the spec
- James exists in the database as a team member
- Excluding specific agents by role adds unnecessary logic
- Having the file does not mean the agent must be dispatched

Additionally, a `codebase-analyst.md` file exists as a utility agent for repository analysis. This was not in the spec but was added as a standalone tool.

**Actual file count:** 10 files (9 roster members + codebase-analyst), vs. the spec's 8.
